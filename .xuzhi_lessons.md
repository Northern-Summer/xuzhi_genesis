## 核心教训（已验证，不可推翻）

### L1：workspace/tmp/ 会被清空
**教训**: 不要把任何需要持久化的东西放在 `/tmp` 或 `~/.openclaw/workspace/tmp/`。
**影响**: 2026-03-22 多次 — autorapatch/、checkpoint.py、agent_watchdog.py、多个 cron 失效。
**正确做法**: 所有持久脚本放在 `~/`（home directory）。

### L2：Gateway 重启后 cron 从内存恢复旧版本
**教训**: OpenClaw cron 存在 Gateway 进程内存里，不是持久化存储。
**影响**: 每次 Gateway 重启，所有旧 cron 全部回来，新修的 cron 丢失。
**正确做法**: 用 `.cron_spec.json` 作为唯一真相来源。Gateway 重启后立即运行 `cron_restore.sh` 从 spec 重建。

### L3：commit = 唯一的保存动作
**教训**: 写完代码不 commit = 没保存。系统崩溃后全部丢失。
**影响**: 2026-03-21~22 多次修复后，每次重启归零，重新做。
**正确做法**: 任何修复后立即 commit + push。不 commit = 未完成。

### L4：没有 checkpoint 的修夏等于没修
**教训**: 修完之后，如果不知道当前状态是什么，下次唤醒我就是从错误状态开始。
**影响**: 新 session 看到的是旧版 cron、旧版脚本、旧版配置。
**正确做法**: 修复完成后更新 `~/.xuzhi_checkpoint.json`，记录当前所有关键状态。

### L5：错误不提炼成原则，就会重复出现
**教训**: 每次错误记录了但没提炼成行为约束，下次遇到同样情况又失败。
**影响**: 同样问题今天出现 6 次。
**正确做法**: 每次错误必须提炼成一句话原则，写进 `~/.xuzhi_lessons.md`，并附上"遇到类似情况应该先做什么"。

### L6：exec session 不可靠，必须有持久化后备
**教训**: exec session 会断线、hung 住、被系统回收。
**影响**: 依赖 exec 的操作（Gateway 重启、cron 管理）可能随时失败。
**正确做法**: 任何需要持久化的操作，必须写成脚本文件 commit，不依赖当前 session 存活。

### L7：systemd 是 Gateway 崩溃后的唯一持久恢复手段
**教训**: Gateway 崩溃后，如果不用 systemd 管理，重启完全依赖手动。
**影响**: 每次崩溃需要人介入，系统无法自我恢复。
**正确做法**: `systemctl --user enable openclaw-gateway` 并验证 `Restart=on-failure`。

## 遇到以下情况应该先做什么（行为约束）

| 情况 | 第一反应 |
|------|---------|
| exec 超时/无响应 | 写日志到 memory/YYYY-MM-DD.md，不继续执行 |
| Gateway 无响应 | 检查 `systemctl --user status openclaw-gateway`，不盲目重启 |
| 发现 cron 数量异常 | 立即对比 `~/.cron_spec.json`，不对就运行 `cron_restore.sh` |
| 任何修复完成 | 立即 commit，不等 |
| 新 session 开始 | Step 0 先读 checkpoint 和 lessons，不做任何操作 |
| 发现 workspace/tmp 变空 | 运行 `cron_restore.sh` 验证 cron 状态 |

## 已发生的错误（按时间顺序）

1. **autorapatch/ 丢失** — 放在 workspace/tmp/，被清空后丢失
2. **15个失效 cron 并行运行** — 多个 cron 指向 workspace/tmp 里的脚本
3. **Gateway 无限重启循环** — 没有检查根因就反复重启
4. ** lessons 没有提炼** — 记录了但没有变成行为约束
5. **checkpoint 没有写** — 修完了没有记录当前状态
6. **新 session 总是从旧状态开始** — 缺少断点恢复机制
