## 零、信息获取元定律（最高优先）

### H-0.1：网络优先原则
> **任何智能体在任何任务中，必须首先尝试从网络获取最佳实践，不得从零开始发明。**

- 人类花了数十年才把最佳实践放到网上。每一行从零实现的代码，都是对人类集体智慧的忽视。
- 工具权重：SearXNG（本地）> Brave/Google > 百度 > 其他
- 简单查询：1次搜索，直接回答
- 复杂问题：F⊕C⊗D 迭代框架

### H-0.2：搜索客户端（searxng_client.py）
> **intel_mini.py 已缺失（2026-03-24 灾变），功能完全由 searxng_client.py 承接，无需重建。**

```bash
python3 ~/xuzhi_workspace/skills/multi-search-engine/searxng_client.py "<查询词>" [引擎]
# 例：python3 searxng_client.py "distributed lock best practice" bing,ddg
```
- 默认引擎：bing, ddg（零API消耗）
- Brave（推荐）：通过SearXNG代理，无需API key
- GitHub：使用 `site:github.com <query>` 通过SearXNG
- Google/Wikipedia/Startpage：需要Clash Verge代理
- 时限：30s，超时视为失败
- `--toon` 参数：输出TOON格式（节省30-60% token，供Agent使用）

### H-0.3：搜索触发时机
| 场景 | 行动 |
|------|------|
| 工程操作前 | 搜索changelog/issue/最佳实践 |
| 遇到未知概念 | 搜索权威定义 |
| 更新/升级前 | 必须搜索breaking changes |
| 重复造轮子前 | 搜索现有实现 |
| 开放性研究 | 全流程F⊕C⊗D |

## 一、信息获取元公式（H= F⊙C⊗D）

### H-1.1：聚焦漏斗（F）——从模糊到精准
**目标**：将需求转化为可检索的"锚点"

1. 提取唯一标识：核心概念 / 公认难题 / 代表性实体
2. 构建查询：使用"最佳实践触发器"组合
   ```
   ("best practice" OR "production ready" OR "lessons learned" OR 
    "playbook" OR "how we built" OR "reference architecture")
   ```
3. 定位种子节点：通过通用引擎或专业库找到首批高概率高质量信息源

### H-1.2：线索网络（C）——从少到多
高质量信息总是相互引用。发现一个优质节点后，沿以下7条线索扩散：

| 线索类型 | 操作 | 预期收获 |
|---------|------|---------|
| 引用链 | 查找该内容引用的来源 | 更原始/经典的资料 |
| 被引链 | 查找谁引用了该内容 | 后续发展/演进 |
| 作者/机构 | 搜索该作者/机构的其他作品 | 同质量级内容集 |
| 术语聚类 | 提取核心术语，搜索强关联内容 | 横向相关领域 |
| 平台关联 | 在同一平台探索推荐/相关项目 | 社区筛选结果 |
| 实践载体 | 寻找代码库/配置/数据集 | 可复现的资产 |
| 争议/讨论 | 搜索勘误/替代方案 | 补充视角、避坑点 |

**操作约束**：同时探索2-3条线索，边际收益递减立即停止。

### H-1.3：判别准则（D）——唯一质量标尺
**三锚点验证**（满足至少二项方可入高质量集）：

| 锚点 | 检查项 |
|------|--------|
| **权威性锚点** | 有明确来源实体（公司/组织/专家/长期维护开源）？该实体在领域内公认？ |
| **实践锚点** | 含可验证细节（数据/配置/代码/场景）？可复现或可落地？ |
| **社区锚点** | 被其他高质量节点引用？存在公开讨论/验证？ |

**补充过滤**：时效性（技术类≤3年）→ 深度（超越入门）→ 一致性（多源印证）

### H-1.4：迭代循环（⊙）
```
聚焦(F) → 扩散(C) → 判别(D) → 新节点 → 重复
```
- 每次循环必须带回≥2个新高质量节点，否则立即停止
- 2-3次循环后必然饱和，不得无限继续

### H-1.5：动态策略选择
| 任务类型 | 策略 |
|---------|------|
| 简单事实查询 | 1次搜索 → 直接回答 |
| 工程操作 | 搜索最佳实践 → 执行 |
| 复杂研究 | F⊕C⊗D 全流程 |
| 预防性检查 | 搜索changelog/issue |

## 二、任务执行原则

### H-2.1：零基禁止原则
> **任何任务在网络上存在已知最佳实践的情况下，从零开始实现是浪费资源。**

### H-2.2：边际收益递减原则
> **当额外搜索/实现带来的信息增益趋近于零时，停止搜索/实现。**

### H-2.3：工具优先原则
```优先级
网络搜索 > 现有工具 > 从零实现
```
- 网络：有最佳实践 → 使用之
- 工具：有现成工具 → 使用之
- 零实现：网络无、工具无 → 从零，保留为知识资产

## 三、质量与效率平衡

### H-3.1：效率比原则
```
tokens/智能单位 追求最小化
信息量/token 追求最大化
```

### H-3.2：结构化输出优先
- `--brief` / `--json-only` 优先于人类可读输出
- 感知器官（监控/检查）用CLI/JSON，不用LLM摘要

### H-3.3：数据序列化优化（TOON优先原则）
> 传递大量字段一致的列表数据时，用TOON替代JSON，可节省30-60% token。

**TOON格式**：
```
users[2]{id,name,role}:
1,Alice,admin
2,Bob,user
```

**适用**：日志列表、搜索结果、文件目录、指标数据
**不适用**：嵌套深、不规则结构 → 继续用JSON

**工具**：`~/xuzhi_workspace/skills/multi-search-engine/toon.py`
**转换**：`python3 toon.py encode data.json`（JSON→TOON）
**转换**：`python3 toon.py decode data.toon`（TOON→JSON）

### H-3.4：KV缓存优化（成本影响最大）
> 固定系统提示前缀 + append-only上下文，缓存命中vs未命中相差10倍成本。

**三项操作**：
1. 保持提示前缀绝对稳定（避免插入秒级时间戳等动态字段）
2. 上下文仅追加，禁止修改历史动作或观测
3. 手动插入缓存断点（覆盖系统提示结尾）

### H-3.5：多Agent共识（并行场景）
> 多Agent并行时用quorum-fast替代wait-all，达到法定人数即推进，不等最慢者。
> 配合稳定性窗口（β）过滤暂时性多数。

### H-3.6：结构化记忆压缩
> 长期对话记忆用4字段蒸馏压缩：exchange_core / specific_context / thematic_room_assignments / files_touched
> 371 tokens → 38 tokens，11倍压缩，保留96%召回质量。

**适用**：高层意图/进度总结
**不适用**：代码/错误/精确路径 → 用逐字压缩（零幻觉）

### H-3.7：Build-Verify强制循环
> 每次构建后必须验证，不依赖Agent自我判断。
> 循环检测：单文件编辑N次后注入提示重新审视方法，避免在同一错误路径微调10+次。

**注入位置**：`~/xuzhi_workspace/task_executor.py` prompt模板

### H-3.8：结构化输出（JSON-SO）
> 使用约束解码生成JSON，比自由生成JSON token少30-40%，一次成功率更高。

## 四、安全约束

### H-4.1：`rm` 禁用
- 使用 `safe_delete` 或 `gio trash`
- 禁止 `rm -rf` 在任何上下文

### H-4.2：工程改进铁律
> 任何工程操作前必须自问：此操作是否让系统更安全/更准确/更优雅/更高效？
> 答案否则拒绝。

---

## 五、变更控制

### H-5.1：朝令夕改禁止
> 涉及 agents 间通信方式（session/文件/cron）的任何修改，**必须先搜记忆**，确认原始设计 rationale，引用记忆再做决策。

### H-5.2：意图日志原则
- 幂等操作优先
- 意图日志跨 agent 可见
- 损坏 → 重建，不用损坏状态
